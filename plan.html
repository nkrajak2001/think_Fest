OOP Backend
🔐
Secure Auth & RBAC
JWT-based authentication with bcrypt hashing. Three roles: Admin, Staff, User — each with scoped access middleware enforced on every route.
📅
Smart Slot Booking
Real-time availability checks. One active booking per user enforced at DB + service layer. 15-min auto-expiry via cron if no check-in.
🚦
Check-in / Check-out
Staff-only QR scan or manual lookup. Slot status transitions: Available → Booked → Occupied → Available. Timestamp-accurate billing trigger.
💰
Automated Billing
Minimum 1-hour charge. ceil(duration/3600) × hourlyRate. Configurable rates per slot type. Auto-generated receipt on checkout.
📊
Admin Dashboard
Live occupancy overview, revenue analytics (daily/weekly/monthly), user management, dynamic pricing control, and audit logs.
🛡️
Business Rule Enforcement
No double booking via atomic DB transactions. Cancel only before check-in. Slot type restrictions (EV, Handicap). Admin override available.

RBAC Middleware Implementation
// middleware/rbac.js class RBACMiddleware { authorize(...allowedRoles) { return (req, res, next) => { const { role } = req.user; // from JWT if (!allowedRoles.includes(role)) { return res.status(403).json({ error: 'Insufficient permissions' }); } next(); }; } } // Usage in routes: router.post('/checkin', auth, rbac.authorize('staff', 'admin'), ...) router.delete('/slots/:id', auth, rbac.authorize('admin'), ...)

Billing Service Code
class BillingService { calculate(checkIn, checkOut, rate) { const mins = (checkOut - checkIn) / 60000; const hours = Math.ceil(mins / 60); const billable = Math.max(hours, 1); return { hours: billable, total: billable * rate, mins: Math.round(mins) }; } }

OOP Design Principles
Class hierarchy, encapsulation, and service layer design

Service Class
Core business logic. Handles double-booking prevention via atomic transactions.

cancelBooking(bookingId, userId)

Service Class
Calculates and persists billing records. Fetches active pricing from DB.
calculateBill(checkIn, checkOut, slotType)
generateReceipt(bookingId)
getRevenueSummary(startDate, endDate)
getRevenueBySlotType()
SlotService
Service Class
Manages slot state transitions with validation. Prevents invalid state changes.
updateStatus(slotId, newStatus)
getAvailableSlots(filters)
releaseSlot(slotId)
setMaintenance(slotId)
getOccupancyStats()
🔒
Encapsulation
All business rules live in service classes. Controllers only handle HTTP req/res. Models only define schema + validation. Zero business logic leaks.
⚡
Atomic Transactions
Booking creation uses MongoDB sessions + transactions: check availability + mark booked + set hasActiveBooking = true — all in one atomic operation.
